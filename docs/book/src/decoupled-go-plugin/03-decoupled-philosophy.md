# Chapter 3 — Decoupled Scaffold Philosophy

## The Core Problem

The standard `go/v4` plugin generates scaffolding and then expects the user
to add business logic **inside the same files**. For example, you add your
reconcile logic inside `internal/controller/<kind>_controller.go`, right next
to the generated boilerplate.

This coupling creates a painful upgrade cycle:

```
Upgrade go/v4 →  regenerate files →  merge conflicts →  manual reconciliation
```

Every upgrade becomes a laborious diff/merge exercise, discouraging users
from keeping their dependencies up-to-date.

---

## The Solution: Two Distinct Layers

`decoupled-go/v1` enforces a strict boundary between two layers:

### Layer 1: Platform Layer (plugin-owned)

```
internal/platform/
```

This directory contains **all generated framework code**:
- Manager setup and scheme registration
- Generated reconciler shell (calls user hooks via interfaces)
- Generated webhook stubs (call user validators via interfaces)
- Any generated wiring or adapter code

**Rules:**
- Every file has a `DO NOT EDIT` header with provenance info.
- Files are **fully overwritten** on every `regenerate` run.
- Users must not edit these files.

### Layer 2: App Layer (user-owned)

```
internal/app/
```

This directory contains **all user business logic**:
- Hook implementations (reconcile, finalize, validate, default)
- Domain types and helpers
- Any custom extension modules

**Rules:**
- Files are created **once** by the plugin (as stubs) and never touched again.
- Users own 100% of these files.
- The plugin uses `if-not-exists` semantics: create only if the file is absent.

---

## File Ownership Model

### Provenance Header

Every platform-layer file begins with a provenance header:

```go
// Code generated by decoupled-go/v1 (v1.0.0). DO NOT EDIT.
// Source: kubebuilder init --plugins decoupled-go.example.com/v1
// Regenerate: kubebuilder edit --plugins decoupled-go.example.com/v1 --regenerate
```

This makes it visually obvious which files are managed by the plugin, and
it conveys how to regenerate them.

### The Scaffold Lock File

The plugin creates a lock file at `gen/kb/decoupled-go.v1/.kb-scaffold-lock.yaml`
that records:

```yaml
# Scaffold lock — maintained by decoupled-go/v1. DO NOT EDIT.
pluginVersion: v1.0.0
pluginKey: decoupled-go.example.com/v1
generatedAt: "2024-01-15T10:30:00Z"
files:
  - path: internal/platform/doc.go
    sha256: abc123...
    zone: platform
  - path: internal/app/myapp/hooks.go
    sha256: def456...
    zone: app
    policy: create-only
```

Zones and policies:

| Zone | Policy | Meaning |
|---|---|---|
| `platform` | always overwrite | Generated code, fully managed |
| `app` | `create-only` | Stub created once, never overwritten |
| `bridge` | marker-driven | Partial management via markers (advanced) |

## Why This Works

### Upgrade Workflow (Post-decoupling)

```
1. bump plugin version in tool config
2. run: kubebuilder edit --plugins decoupled-go.example.com/v1 --regenerate
3. diff only internal/platform/** (no user code mixed in)
4. done — user code in internal/app/** is untouched
```

### The Mental Model

Think of it like a well-designed library vs an application:

- **Platform layer** = the library (you use it, you don't own it)
- **App layer** = your application code (you own it completely)

The library changes its internals; your application code stays the same as
long as the library's public API (the Go interfaces) is stable.

---

## Interface-First Design

The platform layer exposes a set of **stable Go interfaces** that the app
layer implements. As long as these interfaces don't have breaking changes,
the platform layer can be upgraded freely.

```
internal/platform/
└── reconciler/
    └── hooks.go        ← defines ReconcileHooks interface (stable contract)

internal/app/
└── myapp/
    └── hooks.go        ← user implements ReconcileHooks (never overwritten)
```

The generated reconciler in `internal/platform/reconciler/reconciler.go`
does exactly this:

```go
// generated — DO NOT EDIT
type Reconciler struct {
    client client.Client
    hooks  hooks.ReconcileHooks  // ← calls user code via interface
}

func (r *Reconciler) Reconcile(ctx context.Context, req ctrl.Request) (ctrl.Result, error) {
    var obj appsv1.MyApp
    if err := r.client.Get(ctx, req.NamespacedName, &obj); err != nil {
        return ctrl.Result{}, client.IgnoreNotFound(err)
    }

    if err := r.hooks.BeforeReconcile(ctx, req); err != nil {
        return r.hooks.OnError(ctx, &obj, err)
    }

    result, err := r.hooks.Reconcile(ctx, &obj)
    if err != nil {
        return r.hooks.OnError(ctx, &obj, err)
    }

    return r.hooks.AfterReconcile(ctx, &obj, result)
}
```

---

## What Happens on First `create api`

1. Plugin generates `internal/platform/reconciler/<kind>_reconciler.go`
   (managed, overwrites allowed).
2. Plugin creates `internal/app/<kind>/hooks.go` as a stub **only if the
   file does not exist** (create-only).
3. Plugin updates the scaffold lock file.

On subsequent `create api` (or `regenerate`):

1. `internal/platform/reconciler/<kind>_reconciler.go` → **overwritten** with
   the latest template.
2. `internal/app/<kind>/hooks.go` → **unchanged** (user's code is safe).
3. Lock file → updated with new checksums and plugin version.

---

## Compatibility Check on Regeneration

When a user upgrades the plugin version and regenerates, the plugin:

1. Reads the old lock file.
2. Detects interface changes (e.g., a new method was added to `ReconcileHooks`).
3. Prints migration warnings listing which user files may need updating.
4. Regenerates platform files.
5. Writes a new lock file.

This makes upgrades **visible and auditable** rather than silent and error-prone.

---

## Non-Goals

The decoupled philosophy does **not** mean:

- ❌ Users can never look at platform files — they can and should, to understand the plumbing.
- ❌ Users can never fork the plugin — they can for highly custom needs.
- ❌ The plugin is responsible for user's business correctness — the app layer is 100% user responsibility.
